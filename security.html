FERPA

Family Educational Rights and Privacy Act

RoleCall is designed to be FERPA-compliant by architecture, not just by policy. Here's why:

Key Point: RoleCall does not collect, process, or store education records as defined by FERPA (20 U.S.C. § 1232g). No grades, enrollment status, student IDs, or academic history are collected or transmitted.

Why RoleCall does not trigger FERPA

  • No student PII required. Students can use the entire platform without providing their name, email, student ID, or any identifying information.
  • Optional account creation. If a student chooses to create an account (for cross-device sync), only an email address is collected. This is not linked to any institutional student information system.
  • Practice data is device-local. Interview practice scores, session history, bookmarks, and streak data are stored in the browser's localStorage or the mobile app's local storage. This data never leaves the device.
  • No institutional data integration. RoleCall does not connect to any SIS, LMS, or institutional database. There is no data exchange between RoleCall and the institution.
  • No student-level reporting to institutions. Institutional partners receive only aggregate, anonymized usage statistics (e.g., "47 students practiced 312 sessions in March"). No individual student data is shared.

What if a student creates an account?

If a student optionally creates an account using Google SSO or email/password, the following data is stored in Firebase Authentication and Firestore:

  • Email address (from Google SSO or entered manually)
  • Display name (from Google profile, if using SSO)
  • Subscription status (if applicable)
  • Practice data (synced from device-local storage, if the student opts in)

This data is controlled by the student, not the institution. RoleCall acts as a school official only if a signed DPA is in place. Otherwise, RoleCall operates as an independent third-party service that the student chooses to use.

Recommendation: For strictest FERPA compliance, advise students to use RoleCall without creating an account. All core functionality works without authentication. If account creation is desired, a DPA should be executed.
Data Architecture

How Data Flows

Without an account (default)

  1. Student visits rollcallinterviewprep.com
  2. All role data, questions, and tips are loaded from the static site (no user-specific data transmitted)
  3. Practice sessions run entirely in the browser
  4. Scores, history, and bookmarks are saved to localStorage
  5. No data is sent to any server

With an optional account

  1. Student creates an account via Google SSO or email/password
  2. Firebase Authentication stores: email, display name, UID
  3. Student can optionally sync practice data to Firestore
  4. Firestore stores: practice sessions, bookmarks, subscription status
  5. Data is associated with the student's Firebase UID, not a student ID

Infrastructure

  • Hosting: Firebase Hosting (Google Cloud Platform)
  • Authentication: Firebase Authentication
  • Database: Cloud Firestore (when accounts are used)
  • Payments: Stripe (PCI DSS Level 1 compliant)
  • Analytics: Google Analytics (anonymized page views)
  • Domain: rollcallinterviewprep.com (HTTPS enforced)
Encryption

Encryption & Transport Security

✅
HTTPS Everywhere. All pages are served over TLS 1.2+ via Firebase Hosting. HTTP requests are automatically redirected to HTTPS. HSTS headers are enabled.
✅
Firebase Security Rules. Firestore access is restricted by security rules. Each user can only read/write their own data. No public read access.
✅
Stripe PCI Compliance. All payment processing is handled by Stripe (PCI DSS Level 1). RoleCall never stores or transmits credit card numbers.
✅
No plain-text secrets. API keys for Firebase and Stripe are client-side SDK keys (designed for public use with security rules). Server-side secrets (Stripe webhook keys, admin credentials) are stored in environment variables, never in code.
✅
Content Security Policy. CSP headers restrict script sources to known CDN providers (Firebase, Stripe, Google Analytics). Inline scripts are minimized.

Access Controls

Authentication & Authorization

For students (optional)

  • Google SSO (OAuth 2.0) or email/password
  • Email verification required for email/password accounts
  • Password requirements: minimum 6 characters (Firebase default)
  • Students can delete their account and all associated data at any time

For institutions (admin dashboard, when available)

  • Separate admin authentication (not the same as student accounts)
  • Read-only access to aggregate usage data
  • No access to individual student data or practice sessions
  • Admin access is granted manually by RoleCall staff
Data Deletion

Data Retention & Deletion

  • Local data: Students can clear all local data by clearing their browser storage or uninstalling the app. No request needed.
  • Account data: Students can delete their account from the app's Settings page. This permanently removes their authentication record and all synced Firestore data.
  • Institutional data: Aggregate usage reports are retained for 1 year after the pilot/license ends, then deleted.
  • Google Analytics: Data retention is set to 14 months (Google's default).
Important: RoleCall does not currently support bulk data deletion requests (e.g., a school requesting deletion of all student data). Since student data is device-local and not linked to institutions, this is not applicable in most cases. For students with accounts, individual deletion requests can be sent to rollcallinterviewprep@outlook.com and will be processed within 30 days.
